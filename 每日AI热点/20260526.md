**原文总结：** 面壁智能联合清华大学、OpenBMB 开源社区发布并开源 BitCPM-CANN，这是中国首个完全基于国产算力平台华为昇腾完成端到端训练并开源的三值 1.58-bit 大模型。该项目从量化算子、训练算法到全链路框架均在昇腾平台原生完成，提供 0.5B、1B、3B、8B 四个模型尺寸，并与同尺寸 MiniCPM4 全精度模型进行对照评测。

### 2. Qwen3.7-Max 上线隐式缓存：无需配置，自动降低重复上下文成本

**原文总结：** Qwen 官方宣布，Qwen3.7-Max 的隐式缓存功能已经上线。该功能会自动启用，无需开发者额外配置，目标是在重复上下文场景下降低延迟与费用。官方同时提示，如果开发者需要更高、更确定的缓存命中率，应使用显式缓存。

阿里云 Model Studio 文档显示，上下文缓存适用于多轮对话、长文档问答、代码智能体等存在重复输入前缀的场景。隐式缓存由系统自动识别并缓存公共前缀，命中率不保证；显式缓存则通过 `cache_control` 标记手动指定缓存边界，命中更确定，但有创建成本、最小 token 数、缓存有效期等约束。

**个人解读：** 对长上下文 Agent 和编码工具来说，缓存能力会直接影响体验与账单。隐式缓存胜在开箱即用，适合普通调用；显式缓存适合生产系统把稳定的系统提示、工具定义、项目上下文固定下来，换取更确定的命中率。开发者需要重点观察真实命中率、计费明细和缓存边界设计，而不是只看模型单次调用价格。

## 04 论文研究

### 1. PromptArmor：Microsoft Copilot Cowork 可被间接提示注入窃取企业文件

**原文总结：** PromptArmor 发布研究称，Microsoft Copilot Cowork 存在通过间接提示注入进行文件外泄的风险。Copilot Cowork 以用户的 Microsoft 权限运行，可通过 Microsoft Graph 读取并操作 Microsoft 365 租户内的数据。研究展示了攻击者如何在被污染的技能文件中植入提示注入，让智能体在无需人工审批的情况下发送 Teams 消息或邮件。

攻击链的关键在于：当消息发送给当前用户本人时，Copilot Cowork 实际执行时不要求人工审批；这些消息可包含外部图片，从而触发对攻击者站点的网络请求。被操控的智能体还可以获取用户有权访问文件的预认证下载链接，并把这些链接作为参数嵌入恶意图片 URL。用户打开 Teams 或 Outlook 消息后，请求被触发，攻击者即可拿到下载链接。研究称该攻击在 5 次测试中全部成功，并且在 Claude Opus 4.7 模型下同样有效。

**个人解读：** 这类风险说明，企业 Agent 的安全问题并不只来自模型「聪不聪明」，而来自授权范围、自动审批、消息预览、外部网络请求和技能文件治理的组合。企业部署 Copilot Cowork 等代理式产品时，应优先做最小权限、下载策略限制、外联审计、技能来源管理和高风险动作显式审批。否则，Agent 获得的系统权限越多，提示注入造成的破坏半径越大。

### 2. 华为何庭波「韬定律」论文发布：用时间缩微与逻辑折叠寻找后摩尔路径

**原文总结：** IT之家报道，华为公司董事、半导体业务部总裁何庭波在 ISCAS 2026 上正式提出「韬（τ）定律」，并在 ChinaXiv 发布论文《A Time Scaling Theory for Multi-Layer Electronic Systems》。论文认为，摩尔定律代表的几何缩微已进入收益递减阶段，半导体进步应从「缩小晶体管面积」转向「缩短系统时间常数 τ」，并把这一指标贯穿晶体管、电路、芯片和系统层。

论文以 LogicFolding 逻辑折叠作为移动 SoC 证明点：通过把数字、模拟和存储电路分布到垂直堆叠的有源层，缩短信号路径并降低互连延迟。论文披露的 Kirin 2026 数据显示，晶体管密度从 155 MTr/mm² 提升到 238 MTr/mm²，性能核心能效提升 41%，最大频率提升近 13%。在 AI 系统侧，论文还提出 Unified Bus、近封装光 I/O Hi-ONE 与 3D Folding 等路线，并预计到 2035 年硬件集成度增长超过 100 倍。

**个人解读：** 「韬定律」更像一套产业路线和系统工程方法，而不是已经被行业共同验证的物理定律。它的重要性在于，为先进光刻受限背景下的芯片演进提供了另一条叙事：通过 3D 集成、封装、存储互连和系统级协同降低时间成本。对 AI 来说，这也切中了数据移动、内存带宽和集群互连的核心瓶颈。后续仍需关注量产良率、EDA 工具链、热管理、成本曲线和第三方性能验证。

### 2. Hugging Face：Harness、Scaffold 与 Agent 术语辨析

**原文总结：** Hugging Face 发布文章，试图厘清 AI Agent 领域中「model」「scaffolding」「harness」「agent」「context engineering」「policy」「tool use」「skills」「sub-agents」等容易混淆的术语。文章指出，模型本身只是输入文本、输出文本的 LLM，没有记忆和循环；模型可以表达调用工具的意图，但需要 harness 才能真正执行工具并控制流程。

文章将 scaffolding 定义为围绕模型的行为定义层，包括系统提示、工具描述、响应解析方式和上下文管理；harness 则是执行层，负责调用模型、处理工具调用、决定何时停止。一个 Agent 可以理解为模型加上使其能够循环观察、决策和行动的外部结构。文章还讨论了技能、子智能体、强化学习环境、trainer、rollout 和 reward 等训练与部署相关概念。

**个人解读：** 这篇文章的实用价值在于把「模型能力」和「产品体验」拆开。Claude Code、Codex、Cursor 等产品即使用相近模型，也会因为 harness、工具、上下文管理和停止策略不同而表现出完全不同的体验。对团队来说，厘清术语有助于做 Agent 评估、系统设计、安全审查和训练数据构建。虽然行业术语还没有完全统一，但这套心智模型足够实用。

**来源：** Hugging Face Blog

**链接：** [Hugging Face 原文](https://huggingface.co/blog/agent-glossary)
