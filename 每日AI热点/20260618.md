# AI Agent 的下一站：每个智能体都是一个文件夹

过去一年，AI Agent 的 Demo 很多。

它们会打开网页、查资料、写代码、规划任务。演示视频里，Agent 像一个不知疲倦的实习生：你说一句，它就开始干活。

可真正进入企业流程以后，故事往往没那么顺。

它跑到一半断了怎么办？它调用了哪些工具，谁能复盘？它要发邮件、改数据、触达客户，谁来审批？它拿到的 token 和内部接口，怎么保证不会被模型随手暴露？

这就是 Agent 从 Demo 到生产的落差。

一个能聊天的助手，和一个能在公司里稳定干活的 Agent，中间隔着一整套工程系统。

2026 年 6 月 17 日，Vercel 发布了开源 AI 智能体框架 Eve。它最抓人的设计是：**每个 AI Agent，就是一个文件目录。**

这句话听起来朴素，但背后的判断很重：AI Agent 正在从“聊天框里的助手”和“炫技 Demo”，进入一个更工程化的阶段。它要像代码仓库一样组织，像应用一样部署，像团队成员一样协作。

## Agent 终于有了“形状”

Eve 的核心设计叫 filesystem-first，也就是“文件系统优先”。

翻译成人话：你不需要在复杂的注册代码、配置中心和控制台里拼一个 Agent。你只要打开一个目录，看里面有哪些文件，就能知道这个 Agent 是谁、用什么模型、遵守什么指令、会调用哪些工具、懂哪些技能、能连接哪些系统、会出现在哪些渠道。

一个 Agent 目录大致可能长这样：

agent/

- agent.ts
- instructions.md
- tools/
- skills/
- connections/
- subagents/
- channels/
- schedules/

`instructions.md` 放长期指令，`tools/` 放工具，`skills/` 放按需加载的知识，`connections/` 负责连接数据库或第三方 API，`channels/` 让它进入 Slack、Teams、Telegram 等入口，`schedules/` 则让它定时运行。

【配图建议 2：概念解释图】
提示词：信息图风格，展示“一个 AI Agent = 一个文件目录”。中心是一个名为 agent/ 的文件夹，向外展开 agent.ts、instructions.md、tools、skills、connections、subagents、channels、schedules 八个文件或子目录，每个节点配简洁中文标签：模型、指令、工具、技能、连接、子智能体、渠道、定时任务。画面像工程文档插图，清晰、扁平、留白充足，适合公众号正文插图，避免复杂代码细节，不出现真实品牌 Logo。

重点不是“文件夹”有多神奇，而是它给 Agent 找到了一种可读、可复制、可版本管理的组织方式。

过去很多 Agent 是从一个聊天循环开始的：模型输入、模型输出、工具调用，再把结果塞回上下文。Demo 很快能跑起来。但一到真实业务，问题就来了：工具怎么注册？权限怎么管？模型改了怎么测试？任务中断怎么恢复？不同渠道里的上下文怎么统一？

Eve 的回答是：这些不该每个团队都从头写一遍。

就像网页后来需要框架，软件后来需要 Git、包管理和 CI/CD，Agent 发展到今天，也需要自己的工程骨架。

**Agent 不该只是 prompt。它应该是一组可以被审查、部署、追踪和回滚的工程资产。**

## 真正能上生产的 Agent，靠的不是会聊天

Vercel 在 Eve 里内置了六类生产级能力。它们听起来没有“模型又变聪明了”那么刺激，但恰恰决定 Agent 能不能进入真实工作流。

第一是持久执行。真实任务很少是一问一答就结束。Agent 可能要等审批、等外部系统返回、等用户第二天再补充信息。Eve 把会话做成持久工作流，每一步都有检查点，服务重启或任务中断后还能恢复。

第二是沙箱计算。Agent 越有用，越可能写代码、跑脚本、处理文件。但模型生成的代码不能默认可信。Eve 给 Agent 准备隔离沙箱，让它能动手，又不至于直接暴露主系统。

第三是人机审批。这是企业最在意的边界。总结文档可以自动，发客户邮件、改业务数据、触达外部系统，就必须在关键节点停下来，让人确认。成熟的 Agent 不是一路自动到底，而是知道什么时候把决定权交回给人。

第四是安全连接。Agent 要有用，就要连接 GitHub、Slack、Snowflake、Salesforce、Notion，甚至企业内部系统。连接越多，权限越敏感。Eve 把连接也文件化，并通过授权机制处理凭证，让模型能用工具，但不直接看见敏感 token。

第五是多通道接入。同一个 Agent 不一定只活在网页里。研发可能在 GitHub 里用它，销售可能在 CRM 里用它，客服可能在工单系统里用它。Eve 支持把一个 Agent 接到不同渠道里，让它更像一个真正可以进入组织流程的工作者。

第六是追踪与评估。Agent 出错时，最麻烦的不是“它错了”，而是没人知道它为什么错。它读了什么上下文？调用了什么工具？工具返回了什么？哪一步判断偏了？Eve 内置 tracing 和 eval，让 Agent 的运行过程可以被观察，版本变化可以被测试。

这些能力放在一起，其实表达了一个判断：

**Agent 一旦会行动，它就不再只是聊天机器人，而是一段会影响现实流程的软件。**

## 从 prompt 工程，到 Agent 工程

过去两年，很多 AI 应用围着 prompt 转。怎么写系统提示词，怎么拆任务，怎么让模型按格式输出，这些当然重要。

但当 Agent 开始接入真实业务，prompt 只是一部分。更大的问题变成了：如何组织能力，如何隔离风险，如何管理权限，如何观察运行，如何评估版本，如何让多个 Agent 协作。

这就是从 prompt 工程走向 Agent 工程。

Eve 的“一个 Agent 一个目录”，本质上是在把 Agent 变成一种新的软件工程单元。过去一个应用可以放进代码仓库，一个函数可以封装成包，一个服务可以部署和监控。未来，一个 Agent 也应该有目录结构、依赖、工具、测试、日志、权限和责任边界。

这会改变我们理解 AI 应用的方式。

Agent 不再只是产品右下角的一个助手按钮，而可能成为企业内部的一类可组合资源。数据分析 Agent 负责查数和画图，销售 Agent 连接 CRM 和邮件系统，支持 Agent 查文档和工单，路由 Agent 判断任务该交给谁。

它们之间的关系，开始像一个小型组织：有前台，有后台；有通才，有专家；有执行者，也有审批人。区别在于，这个组织不是靠口头协作管理，而是靠文件、权限、日志、评估和部署来管理。

## 对开发者和企业的启发

对开发者来说，Eve 提醒我们：未来写 AI 应用，可能不只是写页面、接口和数据库，还要写一个“会工作”的目录。

这个目录里有角色设定，有工具，有连接，有技能，有子 Agent，也有它能出现的渠道。应用不再只是等用户点击，它会在合适的条件下主动行动。

对企业来说，启发更直接。

很多公司并不缺 Agent 想法，缺的是规模化管理 Agent 的方式。一个团队做一个 Demo 很容易；十个团队各做十个 Agent 以后，工具、权限、日志、知识库和渠道全散了，问题才真正开始。

所以，企业不能只问“这个 Agent 能不能做事”，还要问：它能不能被审计？能不能暂停和恢复？关键动作有没有审批？权限是不是最小化？模型升级后有没有评估？出了问题，能不能追到具体步骤？

这些问题听起来不性感，却决定 Agent 能不能从会议室里的演示，变成业务流程里的基础设施。

## 也别急着把 Eve 当成终局

当然，Eve 还处在早期。

Vercel 官方披露，内部已经运行了上百个 Agent，包括数据分析工具 d0、自动销售代理 Lead Agent、支持智能体 Vertex 等。这说明 Eve 至少在 Vercel 自己的业务里经过了实践，但它能不能成为更广泛的标准，还要看生态、开发者采用度、企业集成能力和长期稳定性。

Agent 框架这个赛道不会只有 Eve。不同云平台、开源社区和企业都会提出自己的抽象。最终留下来的，不一定是最早发布、最会讲故事的那个，而是最能和真实工作流磨合的那个。

但 Eve 的价值在于，它把一个方向说清楚了：

Agent 的下一站，不是更会聊天，而是更像软件。

它需要目录，需要版本，需要测试，需要权限，需要观测，需要部署，也需要和人协作的界面。

未来的 AI Agent，可能不再只是一个聊天窗口。

它会是一个可以被管理的工作目录：里面有它的角色、工具、技能、连接、审批、日志和同事。

你打开这个目录，看到的不只是代码。

你看到的是一个数字员工真正开始上班的样子。
